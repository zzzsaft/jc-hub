# ERP SQL 多轮会话上下文设计

## 目标

ERP SQL Agent 在同一会话中使用最近六条对话和更早轮次的滚动语义摘要解析追问，同时保留可执行的类型安全查询计划。禁止通过具体问句特判维持上下文。

## 上下文模型

- 最近六条用户/助手消息以原文提供给分析 LLM。
- 更早消息压缩为滚动语义摘要，仅保留指标、维度、时间范围、比较周期、筛选、排序、业务口径、分类规则、用户纠正和未解决歧义。
- 不把原始查询结果行或完整 SQL 放入 LLM 会话上下文。
- 查询执行使用经过运行时类型校验的语义快照；审计脱敏副本不能反向作为执行输入。
- 新陈述覆盖旧摘要时记录被覆盖的字段，以最新明确用户陈述为准。
- 摘要不可用时回退到最近六条消息和最后一个有效计划。

## 时间语义

- “今年”默认表示从当年 1 月 1 日到当前日期的半开区间。
- 年同比表示去年 1 月 1 日到去年同日的半开区间。
- 明确月份表示当年该自然月；同比表示上一年同月。
- 结果列标签由已解析时间窗口产生，不使用“当前期/比较期”等模糊标题。

## 数据流

Agent Runtime 在创建本轮用户消息前读取会话历史和最后成功运行的语义快照，构造 `conversationContext`。分析 Planner 把当前问题、最近消息、滚动摘要与上轮计划一起发送给 LLM，LLM 只输出结构化计划。确定性编译器验证计划字段和分类规则后组合 approved metric SQL。成功运行后更新语义快照和滚动摘要。

## 安全和兼容

- 保留现有 Guard、权限、approved metric/dimension 边界。
- 历史上下文不包含原始结果行、SQL、鉴权信息或技术审计字段。
- 旧的脱敏计划对象必须在类型校验时拒绝，不能触发字符串运行时异常。
- 旧响应仍可由 SQL alias 推导列元数据；新响应优先使用时间感知元数据。

## 验收

三轮问题“按产品类别……上个月同比”→“今年平模头总类……”→“只算 6 月份……和去年同比”必须在同一会话成功，第三轮继承销售订单金额、产品类别和分类合并规则，并覆盖时间为当年 6 月及去年 6 月。超过六条消息后，旧口径通过滚动摘要继续生效。
