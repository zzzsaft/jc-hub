# XTools CRM 订单与跟进记录导出设计

## 目标

在不绕过 XTools CRM 的固定 IP、单会话、权限或登录限制的前提下，使用当前账号正常页面所调用的只读后台接口，导出全部订单、订单相关行动记录及每条行动的完整详情。第一阶段保存为本机 JSONL 和 CSV 文件，不写入项目数据库，不下载附件。

## 已确认的数据源

- 订单列表来自 `contract` 模块的 `/system/runmod.xt` 查询入口，页面当前显示约 18,769 条。
- 行动/跟进列表来自 `action` 模块的 `/system/runmod.xt` 查询入口，页面当前显示约 24,341 条。
- 两个列表均使用 `where`、`bwhere`、`page`、`orderf`、`orderv`、`pn` 等分页和筛选参数，页面允许每页最多 30 条。
- 跟进与订单主要通过行动主题中的订单号关联，例如 `261822订单`、`2026-512订单` 或 `261890、261891订单`。
- 订单数据日志需要额外权限，当前账号不可访问，本次导出不尝试绕过。
- 订单详情中的 `/ds/order.xt?sys=&id=...` 链接缺少必要的 `sys` 参数，本次不猜测该参数，也不依赖此链接。

## 方案选择

采用单会话、单线程、限速的后台接口导出器。

未采用的方案：

- 浏览器逐页点击：订单与行动列表合计超过 1,400 页，且每条行动还需要详情请求，速度不可接受。
- 直连 CRM 数据库：没有数据库连接信息和授权。
- 多会话或并发抓取：可能触发系统的单会话和固定 IP 防护，不符合安全边界。

## 架构与数据流

1. 用户在受支持的浏览器中正常登录 CRM。
2. 会话引导步骤从当前页面已生成的只读请求模板中提取临时会话校验值和模块参数，不读取浏览器 Cookie、密码或密码存储。
3. 临时会话材料仅写入操作系统临时目录，文件权限设为仅当前用户可读；不得打印到控制台、错误日志或导出数据中。
4. 导出器串行抓取全部订单列表页，并解析订单结构化字段。
5. 导出器串行抓取全部行动列表页，保留完整列表记录，并筛选主题中包含订单号的行动。
6. 订单号解析器标准化全角符号、空格和常见分隔符，从一个主题中提取一个或多个订单号。
7. 关联器使用标准化订单号进行精确匹配。一条行动可关联多个订单；歧义或无法匹配的行动单独保留，不做模糊猜测。
8. 对已识别的订单跟进行动逐条调用正常页面使用的只读详情入口，解析完整字段、备注、正文、自定义字段和附件元数据。
9. 全部详情抓取完成后生成关联 CSV、未匹配 CSV、错误清单和导出清单。

## 输出文件

每次导出创建独立目录：

`tmp/xtcrm-export/<YYYYMMDD-HHmmss>/`

目录包含：

- `orders.jsonl`：全部订单结构化记录。
- `actions.jsonl`：全部行动/跟进列表记录。
- `action_details.jsonl`：订单相关行动的完整详情。
- `order_followups.csv`：订单与跟进的多对多关联结果。
- `unmatched_actions.csv`：包含订单字样但无法精确匹配的行动。
- `errors.jsonl`：失败阶段、页码或记录 ID、错误分类和可重试状态。
- `checkpoint.json`：当前阶段、下一页、已完成详情 ID、成功数和失败数。
- `manifest.json`：源总数、导出总数、唯一 ID 数、关联数、失败数、开始/结束时间和输出文件 SHA-256。

导出目录必须保持在 Git 忽略范围内。不得将真实 CRM 数据、会话校验值或临时会话文件提交到仓库。

## 字段与可追溯性

订单至少保留：

- CRM 订单 ID
- 订单号
- 主题
- 分类
- 客户 ID（如响应提供）和客户名称
- 开始日期与最晚发货日期（如响应提供）
- 总金额、回款金额、发货金额、开票金额
- 状态
- 所有者
- 附件元数据
- 来源页码和抓取时间

行动列表至少保留：

- CRM 行动 ID
- 类别
- 主题
- 状态与优先级
- 客户 ID（如响应提供）和客户名称
- 开始日期
- 执行人
- 附件标记或元数据
- 来源页码和抓取时间

行动详情保留页面正常显示的全部结构化字段。已知字段使用稳定名称，未知或自定义字段保存到 `extraFields`。详情记录同时保留行动 ID 和脱敏后的来源路径，确保后续可以追溯而不泄露会话参数。

金额以原始文本和标准化十进制定点字符串双份保留，避免浮点误差。页面中的相对日期（如“今天”“昨天”）同时保留原文，并按抓取日期计算标准日期；无法可靠换算时只保留原文并记录警告。

## 订单号关联规则

- 接受纯数字订单号，如 `261822`。
- 接受带年份和短横线的订单号，如 `2026-512`。
- 接受中文顿号、逗号、斜杠、空格等分隔的多订单号，如 `261890、261891订单`。
- 标准化全角数字、全角连接符和多余空白。
- 只进行精确订单号匹配；不使用编辑距离、客户名或金额进行模糊补配。
- 同一行动包含多个有效订单号时，为每个订单生成一条关联行。
- 订单号重复对应多个 CRM 订单时标记为 `ambiguous`，进入未匹配清单供人工复核。

## 运行保护

- 请求并发固定为 1。
- 正常请求之间默认等待 800ms。
- 网络和临时服务错误最多重试 3 次，使用递增等待时间。
- HTTP 重定向到登录页、响应包含登录表单、会话校验错误、权限提示或关键 DOM 结构缺失时立即暂停。
- 会话失效后要求用户正常重新登录，再从检查点继续；不自动重新登录，不绕过验证码、IP 或设备限制。
- 每完成一页列表或一条详情后原子更新检查点。
- JSONL 按唯一记录 ID 幂等追加；恢复运行时跳过已完成 ID。
- CSV 仅在结构化抓取完成后由 JSONL 重新生成，避免中断造成半行或重复行。
- 控制台进度至少显示当前阶段、已处理/总数、成功/失败数、当前页和最终完成状态。

## 错误处理

错误分类至少包括：

- `session_expired`
- `permission_denied`
- `rate_limited`
- `network_error`
- `unexpected_html`
- `parse_error`
- `record_not_found`
- `ambiguous_order_number`

可重试错误写入 `errors.jsonl` 后按策略重试。不可重试或重试耗尽的记录保留失败条目，继续处理下一条；会话失效和结构整体变化属于全局暂停条件，不继续发送请求。

## 验证策略

### 本地自动测试

使用脱敏、人工构造的 HTML 片段测试：

- 订单列表解析和字段标准化。
- 行动列表解析和字段标准化。
- 行动详情及 `extraFields` 解析。
- 单订单号、多订单号、全角符号和无效主题的提取。
- 精确匹配、多对多关联、重复订单号歧义和未匹配输出。
- CSV 引号、逗号、换行和 UTF-8 内容转义。
- JSONL 幂等写入和检查点恢复。
- 登录页、权限提示和异常 HTML 检测。

### 在线冒烟验证

全量运行前仅抓取：

- 2 页订单
- 2 页行动
- 3 条订单相关行动详情

检查字段、唯一 ID、订单号提取、详情解析和关联结果。冒烟验证通过后才允许启动全量导出。

### 全量验收

- 订单导出数与源页面总数一致，或差异全部出现在错误清单中。
- 行动导出数与源页面总数一致，或差异全部出现在错误清单中。
- JSONL 中不存在重复主键。
- 所有识别为订单跟进的行动均有详情成功记录或明确失败条目。
- 每条关联关系均可追溯到订单 ID 和行动 ID。
- `manifest.json` 汇总与各文件实际记录数和 SHA-256 一致。

## 后续数据库阶段

本设计不修改 Prisma schema，也不写 PostgreSQL。结构化文件经过人工核对后，再单独设计数据库表、敏感字段访问控制、导入幂等键和增量同步策略。
