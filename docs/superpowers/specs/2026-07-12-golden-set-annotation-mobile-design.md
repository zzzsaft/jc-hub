# Golden Set 标注工作台：中文任务与手机端设计

## 目标

让不理解英文的标注人员能只依据脱敏、冻结的证据完成 ProductConfigAgent Golden Set 的盲标；在手机上也能连续完成暂存、校验和提交下一条。页面不得展示系统预测、另一标注席位答案、内部 gold 或其他泄漏盲标的信息。

## 已确认的问题

- 证据区直接展示 `package_source_id`、`date_source` 等内部英文键和值，标注人员无法识别真正要判断的业务内容。
- 产品包表单只有一个“产品名称”输入框，无法实践“多个产品请分别添加”，也不能定义产品角色与关系。
- ERP 候选按钮把 `acceptable_identities` 写进了产品包层；产品包数据结构和 ERP 数据结构没有隔离。
- 390px 宽度下页面只是把桌面两栏顺序堆叠，标题折行、证据冗长，提交按钮不固定，无法高效连续标注。

## 方案比较

1. 仅翻译字段名：改动最小，但仍需要人工从原始字段拼出判断对象，也不能修复多项标注和移动端流程。
2. 中文任务卡 + 原文按需展开：先呈现真正待判断内容、关键事实和可选 ERP 身份，原始英文只作为可展开核对证据。推荐；既降低理解门槛，又保持盲标证据可追溯。
3. 自动翻译全部原文：可读性看似更高，但可能改变原始证据语义、增加依赖和运行不确定性，不适用于冻结的 Golden Set 盲标。

采用方案 2。

## 页面与交互

### 共同结构

- 页首显示层级切换、当前任务进度、保存状态和“稍后处理”。
- 证据卡优先展示中文标题与中文字段标签；英文原文和内部来源 ID 收入“查看原始证据”折叠区。
- 只展示标注所需的事实，不展示 `prediction`、另一席位答案或 gold。
- 任何编辑在 1.2 秒后自动保存；保存中、已保存、保存失败及 revision 冲突都明确反馈。提交前校验，成功后加载下一条。

### 产品包标注

- 先选择“可以判断 / 多种合理理解 / 资料不足 / 无法判断”。
- “可以判断”或“多种合理理解”时，至少有一个产品条目；每条包含中文产品名称、型号（可选）、角色、与主产品的关系，并可删除。
- 支持“新增产品”；第一个条目为主产品，其余条目可选平级产品、组件、附件、备件、销售套件或不确定关系。
- 每个条目自动引用当前冻结证据 ID，提交 payload 与 `PackageGold` 保持一致。

### ERP 身份标注

- 先选择“唯一匹配 / 多个合理 / 证据不足 / 无法判断”。
- 将服务端给出的 ERP 候选显示为可点选的中文候选卡（公司、料号、产品名称、是否有 BOM）；点选会写入 ERP 层 `acceptable_identities`。
- 保留“手动补充身份”入口，避免仅限于系统候选；每个身份必须有公司、料号、产品名称和证据引用。
- “唯一匹配”必须选择一项；“多个合理”至少选择两项；证据不足或无法判断时不得伪造身份。

### 手机端

- 以单列任务流呈现，页面两侧边距和控件宽度均基于 `dvw`、安全区与统一 CSS 变量计算。
- 顶部仅保留任务信息与层级切换；原始证据收折，中文待判断内容优先。
- 底部固定操作栏提供“暂存”和“保存并下一条”，避开 `safe-area-inset-bottom`；所有触控目标至少 40px。
- 在 360px、390px、430px 宽度分别截图检查：无横向溢出、按钮不遮挡、长文本可读、底栏不遮挡输入框。

## 模块边界

- `types.ts`：将 Golden Task、产品项和 ERP 身份的表单类型从 `any` 中抽出。
- `utils.ts`：把冻结 source 转换为中文任务卡；集中处理字段标签、空值和候选去重，保持纯函数可测试。
- `hooks/useGoldenSetWorkbench.ts`：负责取任务、草稿自动保存、校验、提交、错误和保存状态。
- `components/`：任务头、证据卡、产品包表单、ERP 身份表单、手机操作栏各自只承担展示和事件回调。
- `service.ts`：保留现有 API 契约；不修改服务端盲标和 revision 语义。

## 数据与错误处理

- 前端将中文任务卡从 `evidence.source` 派生，原始值不改写也不上传。
- 服务器返回的 revision 始终替换本地 revision；发生 409 时提示重新加载，禁止静默覆盖。
- 网络保存失败保留本地编辑内容，并允许再次尝试；提交校验失败不发请求。
- 没有可读业务证据时，显示“资料不足”的明确任务状态，不用内部字段替代。

## 验证

- 为中文证据映射、产品包校验、ERP 选择规则编写先失败后通过的单元测试。
- 为产品包和 ERP 两层各运行一次真实 API 的 A/B 席位模拟：草稿、提交、读取复核队列。
- 前端构建与 lint 通过。
- 用浏览器插件在桌面、360px、390px、430px 截图；完成一次产品包和一次 ERP 身份的实际页面操作，并确认提交后进入下一条。

## 非目标

- 不修改 Golden Set sealed packet、预测、gold 或复核规则。
- 不引入自动翻译服务或新的前端依赖。
- 不实现生产 ERP 写操作。
