# ERP SQL 诊断模式全业务门禁绕过设计

## 背景

现有 ERP Agent 已能把部分复合问题拆成查询图，但路由能力、指标审批、维度桥、预聚合和其他业务语义门禁会在 SQL 执行前拒绝多数财务复合题。当前五题验收中，四题停在 `capability_route_mismatch`，唯一执行成功的销售/库存/未交付问题也暴露了关联键覆盖不足。

本次目标是增加一个默认关闭的诊断模式：让已授权用户的财务复合问题临时越过业务语义门禁，充分复用现有约 4000 个 SQL 模板，必要时使用受控 LLM SQL 补位；Planner 可执行多条相互独立的只读查询，再由 Analyst 和 Reviewer 基于真实结果完成分析。该模式用于发现模板、Planner、关联键和数据覆盖缺口，不改变生产业务规则。

## 开关与适用范围

新增环境变量：

`ERP_SQL_DIAGNOSTIC_BYPASS_ALL_BUSINESS_GATES=true`

- 只有值精确等于小写 `true` 时启用；缺失、空值、`false`、`1`、`TRUE` 均视为关闭。
- 只允许 ERP SQL Agent 主工作流中的财务复合查询使用，不开放给公共 atomic tool、普通单指标查询或其他 Agent。
- 请求仍须通过登录身份、finance 模块访问权限和 Company 数据范围校验。
- 开关关闭时，现有 Router、Planner、Repository、Composer、Guard 和执行器行为保持不变。
- 不修改指标状态、Golden 数据、业务数据库或 ERP 数据。

## 业务门禁与安全边界

诊断模式允许绕过以下业务语义门禁：

- Router capability 与 Planner 场景不匹配；
- capability route mismatch；
- atomic metric 的 approved/draft/enabled 状态；
- 已有指标定义中的业务完整性阻断；
- 预聚合、粒度、维度桥、join key 和 status predicate 的业务审核状态；
- 现有模板目录无法覆盖完整复合计划。

以下安全边界始终保留，不得由该开关绕过：

- 登录身份和 finance 模块访问权限；
- Company 范围隔离；
- 每次数据库调用只能是一条只读 `SELECT`，禁止多语句、写操作、存储过程和 DDL；
- 物理表、物理字段和 SQL 语法校验；
- `TOP`、最大行数、最大查询数量、超时、并发和结果体积限制；
- SQL、调用参数、计划修正、结果状态和 LLM 分析的审计记录。

“单条只读 `SELECT`”是单次数据库调用的约束，不是整个问题只能查询一次。一个复合问题可以拆成多次数据库调用，每次各执行一条独立的受保护 `SELECT`。

## 计划规范化

Planner 输出后、生成查询图前增加最小的诊断计划规范化步骤。它只从用户原问题中确定性恢复明确表达的槽位：

- 时间范围，例如“今年上半年”“最近 3 个月”“6 月份”；
- 数字阈值，例如“毛利率低于 20%”；
- 明确比较方向，例如“最高”“偏低”“回款慢”；
- 明确数量限制，例如“前 5 类”。

规范化不得由 LLM 自由重写整个计划，也不得猜测用户未表达的阈值、客户、产品或业务口径。每次修正必须记录原计划、修正后计划、修正字段和来源文本；结果增加 `diagnostic_plan_normalized` warning。没有明确证据时保留 Planner 原值，并让后续结果说明口径不确定性。

## 查询生成与执行

查询图按业务问题拆成职责单一的步骤。例如销售、库存、未交付、成本和回款分别查询：

1. 优先从现有模板目录中检索与目标指标、维度、时间范围和表结构最接近的模板。
2. 可直接复用的模板继续走现有参数绑定和 SQL Guard。
3. 模板只能覆盖部分问题时，允许组合多个模板，形成多条查询步骤。
4. 确定性 Composer 和模板均无法生成某一步时，才允许 LLM 为该步骤生成一条只读 SQL。
5. LLM 只能看到授权 Company、允许的物理 schema、目标槽位和行数限制；生成 SQL 必须重新通过物理 schema、Company、只读和 Runtime Guard，失败时不得执行。
6. 每个查询步骤独立记录模板来源或 LLM fallback、SQL、耗时、行数和失败原因。

本次不新建通用 RAG 平台或向量数据库。先复用现有模板检索能力并增加最小 fallback；只有后续证据表明模板检索召回率不足时，再单独设计 embedding/RAG 索引。

## 结果关联与多角色分析

查询结果优先使用确定性业务键关联：Company、客户、订单、订单行、产品等。执行器必须记录每一步的总行数、成功关联数、未关联数和所用键。

- 关联覆盖不足时保留未关联数据，不得强行模糊匹配或伪造关系。
- Analyst LLM 接收原问题、规范化计划、查询摘要、受限结果集、关联覆盖和失败步骤，生成数据支撑的结论、原因分析和建议。
- Reviewer LLM 检查结论是否能由查询结果支持，是否遗漏失败步骤、低覆盖关联、单位、时间范围或口径限制。
- Reviewer 可要求 Analyst 删除无依据结论或降级措辞，但不得生成新 SQL、补造数据或修改查询结果。
- Reviewer 未通过时返回部分结果和明确缺口，不输出看似完整的结论。

## 失败、降级与可观测性

- 任一子查询失败不自动抹掉已成功结果；如果成功数据仍能回答部分问题，则返回 partial/estimate，并列出缺失部分。
- 核心数据全部失败、权限失败或安全 Guard 失败时，不调用 Analyst 生成业务答案。
- 使用该诊断开关的所有成功或部分成功结果统一标记为 `estimate`。
- warnings 至少包含 `diagnostic_all_business_gates_bypassed`；发生计划修正时再包含 `diagnostic_plan_normalized`；使用 LLM SQL 时再包含 `diagnostic_llm_sql_fallback`。
- 前端应展示诊断警告、查询步骤数、成功/失败步骤及关键关联覆盖，不把诊断答案显示为正式财务口径。

## 五题验收标准

使用网页真实提交以下五题：

1. 6 月份销售额最高的 5 类产品、客户、毛利率和主要成本。
2. 今年上半年高收入低毛利客户及产品、订单和主要成本项。
3. 最近 3 个月增长最快产品、库存和未交付订单。
4. 6 月份毛利低于 20% 的订单、客户、产品及材料/加工成本归因。
5. 大额订单、回款慢且毛利偏低的客户。

验收关注诊断链路而不是强制伪造完整答案：

- 四条原 `capability_route_mismatch` 财务题必须进入多查询执行链，不得在 Router/Capability 阶段拒答。
- 第 2 题必须执行“今年上半年”范围，第 4 题必须保留 `<20%` 条件。
- 每题展示实际查询步骤、每步数据量、关联覆盖、Analyst 结论和 Reviewer 状态。
- 有数据时输出由数据支撑的答案；缺表、缺字段、零数据或关联不足时输出部分答案和精确缺口。
- 所有 SQL 均须为独立只读 `SELECT` 并通过保留的安全边界。

## 测试与验证

按用户确认不采用 TDD，先实现后补验证：

1. 开关解析和默认关闭回归。
2. 未授权、非 finance、错误 Company、写 SQL、多语句和未知物理字段仍 fail-closed。
3. 财务复合题可绕过业务门禁并执行多条独立 `SELECT`。
4. 计划规范化覆盖“上半年”“最近 3 个月”“6 月”“低于 20%”和 Top N，并验证修正审计。
5. 模板优先、LLM SQL fallback 次序及 fallback 后完整 Guard。
6. 部分失败、关联不足、Reviewer 驳回和 warning/estimate 状态。
7. 运行 ERP SQL Agent 目标回归、`build:server`、`build:web`。
8. 从网页重新执行五题，保存每题计划、SQL 数、行数、关联覆盖、最终答案或失败原因。

## 非目标

- 不关闭认证、finance 权限、Company 隔离或数据库只读安全。
- 不允许一条数据库调用执行多条 SQL。
- 不审批指标、不修订正式业务口径、不写业务数据。
- 不让 Analyst 或 Reviewer 自行查询数据库。
- 不把诊断模式直接作为生产默认模式。
- 不在本次建设通用知识库、向量数据库或新的 Agent 框架。
